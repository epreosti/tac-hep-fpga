Originally, the three stages (read, blur, write) ran one after another, so the design took 445 cycles total with no pipelining. Resource usage small: 3 BRAMs, 4 DSPs, 100 FFs, and 461 LUTs.
After optimization, the stages run in parallel using DATAFLOW, which cuts latency from 445 to 207 cycles and reduces the interval to 74 cycles. PIPELINE with II=1 allows a new iteration every clock cycle, while ARRAY_PARTITION lets the blur operation read all needed values at once. INLINE off keeps the functions separate so DATAFLOW can work correctly, and STABLE removes unnecessary synchronization overhead.The optimized version is much faster, but it uses more hardware resources (FFs increase from 100 to 820 and LUTs from 461 to 1187) since multiple hardware blocks are running simultaneously.
